94      * If there are any ongoing transactions when a cache is stopped, Infinispan waits for ongoing
95      * remote and local transactions to finish. The amount of time to wait for is defined by the
96      * cache stop timeout. It is recommended that this value does not exceed the transaction timeout
97      * because even if a new transaction was started just before the cache was stopped, this could
98      * only last as long as the transaction timeout allows it.
99      */

116    /**
117     * Only has effect for DIST mode and when useEagerLocking is set to true. When this is enabled,
118     * then only one node is locked in the cluster, disregarding numOwners config. On the opposite,
119     * if this is false, then on all cache.lock() calls numOwners RPCs are being performed. The node
120     * that gets locked is the main data owner, i.e. the node where data would reside if
121     * numOwners==1. If the node where the lock resides crashes, then the transaction is marked for
122     * rollback - data is in a consistent state, no fault tolerance.
123     *
124     * @deprecated starting with Infinispan 5.1 single node locking is used by default
125     */

275    /**
276     * Before Infinispan 5.1 you could access the cache both transactionally and
277     * non-transactionally. Naturally the non-transactional access is faster and
278     * offers less consistency guarantees. From Infinispan 5.1 onwards, mixed
279     * access is no longer supported, so if you wanna speed up transactional
280     * caches and you're ready to trade some consistency guarantees, you can
281     * enable use1PcForAutoCommitTransactions. <p/>
282     *
283     * What this configuration option does is force an induced transaction,
284     * that has been started by Infinispan as a result of enabling autoCommit,
285     * to commit in a single phase. So only 1 RPC instead of 2RPCs as in the
286     * case of a full 2 Phase Commit (2PC).
287     * <p/>
288     * <b>N.B.</b> this option should NOT be used when modifying the
289     * same key from multiple transactions as 1PC does not offer any consistency
290     * guarantees under concurrent access.
291     */
292    public boolean use1PcForAutoCommitTransactions() {
293       return use1PcForAutoCommitTransactions.get();
294    }

312    @Override
313    public boolean equals(Object obj) {
314       if (this == obj)
315          return true;
316       if (obj == null)
317          return false;
318       if (getClass() != obj.getClass())
319          return false;
320       TransactionConfiguration other = (TransactionConfiguration) obj;
321       if (attributes == null) {
322          if (other.attributes != null)
323             return false;
324       } else if (!attributes.equals(other.attributes))
325          return false;
326       if (recovery == null) {
327          if (other.recovery != null)
328             return false;
329       } else if (!recovery.equals(other.recovery))
330          return false;
331       return true;
332    }

334    @Override
335    public int hashCode() {
336       final int prime = 31;
337       int result = 1;
338       result = prime * result + ((attributes == null) ? 0 : attributes.hashCode());
339       result = prime * result + ((recovery == null) ? 0 : recovery.hashCode());
340       return result;
341    }
